# Software Deployment (CI/CD, Deployment Strategies, Containerization, Monitoring, Infrastructure as Code)

This document explains deployment and operations practices for the auction module.

## CI/CD in the Auction Module

Current CI workflow:
- [` .github/workflows/ci.yml`](../.github/workflows/ci.yml)

What it currently does:
- Trigger on push/pull request for key branches.
- Run Rust tests.
- Upload test logs as artifacts.
- Run SonarCloud scan (when token is available).

Purpose:
- Prevent regressions from merging into shared branches.
- Provide auditable quality history via CI artifacts.

## Containerization

Container build support is present via:
- [`Dockerfile`](../Dockerfile)

Purpose:
- Standardize runtime packaging.
- Reduce environment drift between local and deployed runtime.

## Runtime Configuration and Service Composition

Startup composition:
- [`src/main.rs`](../src/main.rs)
- [`src/server.rs`](../src/server.rs)

Key runtime behaviors:
- database connection and migration execution
- scheduler startup (auction closure and outbox publishing)
- optional gRPC client preference with HTTP fallback for wallet/catalog

## Monitoring and Observability

Metrics endpoint and APDEX instrumentation:
- [`src/http/router.rs`](../src/http/router.rs)

Exposed examples:
- request totals and error totals
- APDEX counters and derived APDEX score
- latency histogram buckets
- endpoint-level counters

Dashboard artifact in project infrastructure:
- [`bidmart-auction-service.json`](../../bidmart-infrastructure/monitoring/grafana/dashboards/bidmart-auction-service.json)

## Deployment Strategy Notes

The module currently supports standard pipeline-driven deployment and background worker scheduling.
For advanced operational strategies (blue/green, canary, automatic rollback), those controls are usually implemented at infrastructure/platform level and therefore is documented in the infrastructure repository linked by the team.

## Infrastructure as Code Context

Auction service repository contains application-level deployment assets (Dockerfile, CI workflow, migration script).
Full environment provisioning IaC is expected to live in infrastructure repository/workspace and should be cross-referenced from this document when presenting end-to-end deployment architecture.
