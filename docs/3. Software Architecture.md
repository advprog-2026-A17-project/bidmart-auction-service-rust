# Software Architecture (Architecture, Concurrency, Asynchronous, High-Level Networking)

This document explains architectural decisions implemented in the auction module.

## High-Level Architecture

The auction module uses a layered architecture with explicit boundaries:

- HTTP boundary (`router` + DTO mapping)
- Application orchestration (`AuctionService`)
- Domain model (`ListingAuctionSession`)
- Persistence repositories
- External adapters (catalog/wallet clients)
- Asynchronous event publication (outbox + scheduler publisher)

Main composition point:
- [`src/server.rs`](../src/server.rs)

## Concurrency Architecture

The module uses multiple safeguards to preserve highest-bid correctness:

1. In-process per-auction lock
- Prevents concurrent mutation races within a service instance.
- Implemented in `auction_bid_guard` in [`src/service/auction_service.rs`](../src/service/auction_service.rs).

2. Persistence-level monotonic update
- Update query only promotes higher bids, protecting against stale overwrites.
- Implemented in listing update query path in [`src/service/auction_service.rs`](../src/service/auction_service.rs).

3. Deterministic winner ordering
- SQL ordering: amount desc, time asc, id asc.
- Implemented in [`src/persistence/repositories.rs`](../src/persistence/repositories.rs).

4. Concurrency regression tests
- Competing bids and multi-instance behavior:
  - [`tests/bid_concurrency_tests.rs`](../tests/bid_concurrency_tests.rs)

## Asynchronous Architecture

The module uses outbox-driven asynchronous integration:

- Synchronous path:
  - validate listing state
  - wallet hold/release/convert
  - persist auction/bid state
- Asynchronous path:
  - persist domain event to outbox
  - scheduler publishes outbox events to relay/RabbitMQ

Code references:
- Event creation in [`src/service/auction_service.rs`](../src/service/auction_service.rs)
- Outbox polling/publishing in [`src/scheduler/outbox_scheduler.rs`](../src/scheduler/outbox_scheduler.rs)
- RabbitMQ mapping/publisher in [`src/scheduler/rabbitmq_outbox_publisher.rs`](../src/scheduler/rabbitmq_outbox_publisher.rs)

This reduces coupling between transaction-critical bidding and downstream consumers.

## High-Level Networking Choices

- Public boundary: REST endpoints for gateway/frontend compatibility.
- Internal service calls: optional gRPC clients for catalog/wallet, with HTTP fallback.

Code:
- Router endpoints in [`src/http/router.rs`](../src/http/router.rs)
- gRPC/HTTP client composition in [`src/server.rs`](../src/server.rs)

### High-Level Networking Behavior

- North-south traffic (frontend/gateway to auction service): REST JSON APIs.
- East-west service calls:
  - Catalog lookup for listing eligibility.
  - Wallet hold/release/convert for financial correctness.
- Event propagation to downstream consumers:
  - domain events emitted to outbox then relayed asynchronously.

This split keeps user-facing API simple while allowing transport optimization for internal calls.

## Architectural Extension Implemented

Proxy bidding architecture includes dedicated persisted proxy state:

- table: `proxy_bids`
- repository: `ProxyBidRepository`
- auto-counter logic in `AuctionService`

This allows deterministic, reproducible proxy behavior under concurrent conditions instead of ephemeral in-memory proxy state.

## Architecture Correctness Under Concurrency

The bidding path combines three layers of protection:

1. In-process lock per auction ID.
2. Deterministic winner ordering in persistence queries.
3. Monotonic listing highest-bid update constraints.

Together, these ensure:

- highest bid does not move backward under race.
- equal-value race conditions settle deterministically.
- side effects (outbox + wallet) follow persisted state transitions.

Concurrency behavior is validated by tests that simulate:

- racing low/high bids
- multi-instance race on same auction
- proxy + standard bid interactions
- concurrent proxy bidders with deterministic final outcome

Reference:
- [`tests/bid_concurrency_tests.rs`](../tests/bid_concurrency_tests.rs)

## Asynchronous Consistency Model

The architecture uses eventual consistency for cross-module notifications:

- Bid transaction remains synchronous for correctness-critical state changes.
- Integration notifications (`BidPlaced`, `AuctionCreated`, `AuctionEnded`) are asynchronously delivered.
- Outbox persistence decouples transaction completion from broker availability.

This avoids blocking critical bid placement on external queue reliability while preserving eventual propagation.

## Operational Architecture Notes

- SQLite is used in tests/local for simplicity.
- Production path is designed for PostgreSQL-style deployment with stronger transaction semantics.
- Outbox pattern supports eventual consistency and replay-friendly integration.
