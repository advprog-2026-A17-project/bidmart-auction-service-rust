# Software Quality (Clean Code, Secure Coding, Testing, Profiling)

This document explains quality practices implemented in the auction module.

## Clean Code Practices

- Consistent error modeling with explicit domain/application errors:
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)
  - [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)
- Separation of concerns by file/module boundaries (router/service/domain/repository/client/scheduler).
- Naming and typed value objects (`Money`, `UnixSeconds`, status enum) improve readability and reduce misuse.
- Rule decomposition (`bid_policies.rs`) keeps `place_bid` logic understandable and test-focused.

- A dedicated before/after profiling report documents bottlenecks and optimizations:

  - flame graphs
  - call trees

### Critical Function Profiled

The critical function is bid placement logic on high-contention auctions, especially operations around:

- minimum required bid evaluation
- anti-sniping extension checks
- highest bid update operations

Profiling findings identified heavy cloning/allocation overhead in the naive implementation and guided optimization to reduce this cost significantly.

### Optimization Follow-up

Profiling was not only recorded; it was used to drive code changes:

- reduce unnecessary data cloning
- simplify hot-path data access for highest bid decisions
- align domain representation and logic to reduce allocation-heavy operations

The profiling report includes before/after measurements and visual artifacts (flame graph, call tree, stack chart).

### Optimization Implemented After Profiling (Detailed)

The following optimizations were applied directly based on profiler hotspots:

1. Replace repeated linear highest-bid scans with direct highest-bid tracking.
- Before: naive flow repeatedly searched through bid collections on each bid placement.
- After: domain state keeps a direct `current_highest` reference and computes minimum next bid in O(1).
- Location:
  - [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)

2. Reduce allocation-heavy operations in bid hot path.
- Before: more cloning/string-heavy operations in critical bidding loops.
- After: tighter value-object usage (`Money`, `UnixSeconds`, typed status enum) and reduced cloning overhead in frequent logic paths.
- Location:
  - [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)

3. Tighten state transition logic around anti-sniping and bid acceptance.
- Before: broader state mutation paths with higher compute overhead in naive profile harness.
- After: centralized, explicit transition logic with minimal operations per accepted bid.
- Location:
  - [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)

4. Add deterministic proxy auto-counter logic while preserving concurrency safety.
- Before: proxy handling existed as simple immediate-minimum placement.
- After: persisted proxy maxima with deterministic auto-counter resolution and race-safe application path.
- Location:
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)
  - [`src/persistence/repositories.rs`](../src/persistence/repositories.rs)
  - migration: [`migrations/20260428000000_init.sql`](../migrations/20260428000000_init.sql)

5. Validate improvement using before/after profiling artifacts.
- Evidence:
  - [`docs/PROFILING_REPORT.md`](./PROFILING_REPORT.md)
  - `docs/profile_bidding_naive/*`
  - `docs/profile_bidding/*`

Summary outcome from the profiling report:
- significant reduction in per-bid latency and total execution time in the profiled bidding workload,
- with hotspot shifts from allocation/cloning dominance toward actual business logic execution.
