# Software Design (SOLID Principles, Maintainability, Design Patterns)

This document explains how software design is applied in the auction module implementation.

## Design Structure in This Service

The auction module is organized into layered responsibilities:

- Transport/API layer: [`src/http/router.rs`](../src/http/router.rs), [`src/http/dto.rs`](../src/http/dto.rs)
- Application/service orchestration: [`src/service/auction_service.rs`](../src/service/auction_service.rs)
- Domain rules and entities: [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)
- Persistence layer: [`src/persistence/repositories.rs`](../src/persistence/repositories.rs), [`src/persistence/models.rs`](../src/persistence/models.rs)
- External service adapters (catalog/wallet): [`src/client`](../src/client)
- Async integration boundary (outbox + schedulers): [`src/scheduler`](../src/scheduler)

This separation keeps business rules testable without HTTP/network dependencies and keeps I/O concerns out of domain objects.

## SOLID in the Auction Module

### Single Responsibility Principle (SRP)

- `ListingAuctionSession` only handles auction domain behavior (state transitions, bid validation, anti-sniping, outcome).
- `AuctionService` coordinates use cases (create bid, close auction, persist, publish events, call wallet/catalog).
- `Repository` types only handle SQL/persistence operations.
- `router.rs` only maps HTTP requests/responses and error translation.

### Open/Closed Principle (OCP)

- Auction-type behavior is opened via strategy selection in [`auction_strategy.rs`](../src/service/auction_strategy.rs).
- Bid validation is split into policies in [`bid_policies.rs`](../src/service/bid_policies.rs), allowing extension without rewriting all bidding flow logic.

### Liskov Substitution Principle (LSP)

- External dependencies are consumed through traits (`WalletClient`, `CatalogClient`), allowing HTTP and gRPC implementations to be substituted without changing service logic.

### Interface Segregation Principle (ISP)

- Clients are small, role-focused interfaces:
  - wallet methods for hold/release/convert
  - catalog method for listing summary lookup
- The service depends only on the methods needed for auction behavior.

### Dependency Inversion Principle (DIP)

- `AuctionService` depends on abstractions (`dyn WalletClient`, `dyn CatalogClient`) rather than concrete transport implementations.
- This is visible in constructor variants (`new_with_wallet`, `new_with_catalog`, `new_with_clients`).

## Design Patterns Used

### 1. Strategy Pattern

Used for auction type behavior resolution.

- Where:
  - [`src/service/auction_strategy.rs`](../src/service/auction_strategy.rs)
  - Called from `create_auction` in [`auction_service.rs`](../src/service/auction_service.rs)
- Why:
  - Keep type-specific behavior extensible (`ENGLISH`, future variants) without `if/else` sprawl across service logic.

### 2. Policy Pattern

Used to decompose bidding validation into independent rule sets.

- Where:
  - [`src/service/bid_policies.rs`](../src/service/bid_policies.rs)
  - Invoked in `place_bid_internal` in [`auction_service.rs`](../src/service/auction_service.rs)
- Policies:
  - `TimeBidPolicy`
  - `IdentityBidPolicy`
  - `AmountBidPolicy`
  - `WalletBidPolicy`
- Why:
  - Make each rule testable and modifiable in isolation.

### 3. Repository Pattern

Used for persistence abstraction and query encapsulation.

- Where:
  - [`ListingAuctionSessionRepository`](../src/persistence/repositories.rs)
  - [`BidRepository`](../src/persistence/repositories.rs)
  - [`OutboxRepository`](../src/persistence/repositories.rs)
  - [`ProxyBidRepository`](../src/persistence/repositories.rs)
- Why:
  - Isolate SQL from service logic and improve maintainability/testability.

### 4. Outbox Pattern

Used for reliable asynchronous event publication.

- Where:
  - Event insert in [`auction_service.rs`](../src/service/auction_service.rs)
  - Poll-and-publish in [`src/scheduler/outbox_scheduler.rs`](../src/scheduler/outbox_scheduler.rs)
- Why:
  - Avoid losing domain events when downstream systems are temporarily unavailable.

### 5. Adapter Pattern

Used for external wallet/catalog integration via interchangeable transport implementations.

- Where:
  - trait interfaces under [`src/client/mod.rs`](../src/client/mod.rs)
  - HTTP adapters under [`src/client/http_service_client.rs`](../src/client/http_service_client.rs)
  - gRPC adapters under [`src/client/catalog_client.rs`](../src/client/catalog_client.rs) and [`src/client/wallet_client.rs`](../src/client/wallet_client.rs)
- Why:
  - keep business service independent from transport specifics and serialization mechanics.

### 6. Facade-Orchestrator Style (`AuctionService`)

`AuctionService` acts as a use-case facade over domain, persistence, clients, and outbox publishing.

- Where:
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)
- Why:
  - keep controller/router thin while centralizing transaction flow and invariant enforcement.

## Maintainability Choices

- Strong domain types (`Money`, `UnixSeconds`, status enum) reduce primitive misuse.
- Deterministic ordering and explicit error types make behavior predictable.
- Concentrated orchestration in `AuctionService` with rule decomposition into policies keeps complexity manageable.
- Broad tests (`domain`, `service`, `api`, `concurrency`, `integration`) protect refactoring safety.

## Scalability-Oriented Design Choices

- Separation between synchronous critical path and asynchronous side effects:
  - critical path: validate -> hold funds -> persist bid/listing
  - side effects: outbox event emission and scheduler-based publication
- Explicit concurrency control in service and persistence paths:
  - per-auction lock for in-process contention
  - monotonic listing updates to avoid stale overwrites under race
- Proxy bid state persisted in dedicated table (`proxy_bids`) instead of in-memory cache.
- Support for multiple deployment topologies:
  - same-process local mode for development/testing
  - multi-instance production mode with DB-backed consistency constraints.

## Why This Design is Maintainable Over Time

- New auction types can be introduced behind strategy boundaries without redesigning routing or persistence APIs.
- Validation changes can be localized in policy modules with targeted tests.
- External service transport (HTTP/gRPC) can evolve without touching core bidding domain code.
- Operational concerns (publishing retries, queue transport) are isolated in scheduler/publisher modules.

## Maintainability Considerations from "BidMart: The Future"

The future-direction document highlights that BidMart is expected to evolve beyond classic monetary English auctions. This creates a design requirement: extension must be additive and localized, not a rewrite of the current codebase.

### Future variants explicitly mentioned

- Scholarship-style flow:
  - bounded by time, but winner determination is based on social condition and commitment evaluation by experts, not monetary amount.
- National multi-slot/regional flow:
  - multiple winners per region/capacity, with demand pressure and regional constraints.
- Enterprise auction flow:
  - restricted participants, complex assets, large-value context, and potentially no wallet hold requirement.

### How current design supports those variants

1. Strategy boundary for auction behavior  
- Auction-type routing already exists and is separate from transport/controller logic.  
- This enables adding new winner-selection and validation behavior per auction type while keeping API and orchestration stable.  
- Code:
  - [`src/service/auction_strategy.rs`](../src/service/auction_strategy.rs)
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)

2. Policy decomposition for rule replacement  
- Time, identity, amount, and wallet checks are not hard-coded as one giant block.  
- New variants can replace or bypass selected policies (e.g., enterprise mode may disable wallet hold policy; scholarship mode may replace amount policy with eligibility/evaluation policy).  
- Code:
  - [`src/service/bid_policies.rs`](../src/service/bid_policies.rs)

3. Domain and orchestration separation  
- Domain state machine and bidding semantics are isolated from HTTP and SQL details.  
- This makes non-price winner models safer to add without leaking transport concerns into business rules.  
- Code:
  - [`src/listing_auction_session.rs`](../src/listing_auction_session.rs)
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)

4. Async event boundary for cross-module evolution  
- Outbox events decouple critical auction transaction completion from downstream consumers.  
- As the business introduces more post-bid/post-close workflows, new consumers can be added without blocking the core bid path.  
- Code:
  - [`src/service/auction_service.rs`](../src/service/auction_service.rs)
  - [`src/scheduler/outbox_scheduler.rs`](../src/scheduler/outbox_scheduler.rs)

### Practical extension path

- Scholarship type:
  - add strategy that finalizes outcome from expert-evaluation input rather than highest bid amount.
- Multi-slot regional type:
  - add strategy with per-region capacity and multi-winner selection.
- Enterprise type:
  - add strategy with participant restrictions and wallet-hold policy override.

This keeps the module aligned with the principle that software must evolve continuously with changing business models while preserving code clarity and testability.
